## 1. 워크플로우 자체 평가

### 1-1. 잘 작동한 것

- 4종 .md 세트 (세션로그 / 에러로그 / 의사결정 / 체크리스트)
  - 연습용이긴 하지만, 프로젝트 규모에 비해 과하다 생각할 정도로 상세했다.
- 아키텍처 사전 설계
  - 솔찍히 웹 프로그래밍 쪽으로는 무지한 부분이 있었는데, 알아서 척척 하는 느낌.

### 1-2. 아쉬운 것

- 3자 분담 구조 (나 / Claude Desktop / Claude Code)
  - 생각보다 잘 되지 않았다. 작업자 역할을 맡는 Claude Code 에게 설계 방향성을 묻기도 했고, Claude Desktop 의 Cowork 는 디렉터리에 접근을 할 수 있어 작업을 하기도 했다. 아무래도 하고 싶은 것을 먼저 말하지 않기 때문에 프롬프트가 모호했나보다.

## 2. 다음 프로젝트에 적용할 개선점

> 다음 프로젝트: 유니티. 이번 사이클의 1번 평가에서 나온 문제를 규칙으로 변환한다.

### 2-1. 분담을 "도구"가 아니라 "작업 단계"로

도구(Desktop / Code) 기준으로 역할을 나눈 게 무너진 이유는, 두 도구 모두 양쪽을 다 할 수 있어서 경계가 본인의 규율에만 달려 있었기 때문이다.

- 매 세션 시작 시 노트 맨 위와 AI 첫 메시지에 1줄 선언: **"지금 [설계 / 구현 / 디버깅 / 회고] 단계"**
- 같은 단계 안에서는 어느 도구를 쓰든 모드가 유지된다
- "하고 싶은 것을 먼저 말하지 않음" → "프롬프트가 모호" 문제를 구조적으로 차단

### 2-2. Unity 특수: 세션 노트를 3구획으로

Unity는 AI가 만지지 못하는 영역(Scene, Prefab, Inspector, 플레이테스트)이 크다. 회고가 "코드 작업"만 다루지 않도록 세션 노트를 세 칸으로 나눈다.

- **코드 작업** — AI 협업이 활발한 영역
- **Editor 작업** — 본인이 손으로 한 것 (Scene / Prefab / Inspector)
- **플레이테스트** — 빌드 후 실제 동작 결과

플레이테스트 칸이 비어 있으면 그 세션은 미완. (React 프로젝트에서는 검증을 scope-out 했지만, 게임은 돌아가지 않으면 의미가 없다.)

### 2-3. 문서 통합 — README + DEVLOG

4종 .md 세트를 2종으로 줄인다.

- `README.md` — 개요 / 설계(아키텍처 포함) / 로드맵(체크리스트)
- `DEVLOG.md` — 최신 결정 5개 인덱스 + 세션 로그 링크 모음
- 의사결정은 DEVLOG에 시간순. "현재 설계의 일부"로 굳어지면 README 설계 섹션으로 승급

### 2-4. 세션 노트 분리 + DEVLOG는 인덱스

- 세션 노트: `devlog/YYYY-MM-DD-N.md` (세션당 1파일)
- `DEVLOG.md`는 각 세션의 한 줄 요약 + 링크만
- 에러는 세션 노트 안에 인라인으로 두고, DEVLOG에 "재발 가능 함정" 칸만 따로 모음
